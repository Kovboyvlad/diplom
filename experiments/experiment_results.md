# Результаты экспериментов

Документ фиксирует изменения системы, проведённые эксперименты и выводы.
Модель во всех экспериментах — Gemini Flash и GPT-5. Кейсы — `01_library` и `00_process_sale`.

---

## Хронология изменений

### 1. Одиночный агент (baseline)

**Что сделано:** добавлен режим `run_single_agent()` — один агент-кодер без Analyst и Architect.
Читает требования напрямую и генерирует PlantUML. Сохраняется в `meta.json` как `pipeline_mode: "single"`.
В UI добавлен radio-переключатель между режимами.

**Зачем:** нужна контрольная точка для сравнения — насколько мультиагентный пайплайн
лучше простой генерации одним агентом.

---

### 2. Английские тест-кейсы

**Что сделано:** создана папка `experiments/cases_natural_en/` — все 6 кейсов переведены на английский.

**Зачем:** метрика `entity_coverage_pct` работает через bag-of-words совпадение слов из требований
с именами в PlantUML-коде. Русские требования → английские имена классов → coverage = 0%.
На английских требованиях метрика работает корректно.

---

### 3. Исправление метрики изолированных узлов

**Проблема:** regex для определения стрелок не включал `..|>` (реализация интерфейса).
Все интерфейсы подключённые через `..|>` считались изолированными узлами.

**Исправление:** добавлен паттерн `\.\.\\|>` в `_ARROW_RE` и в `all_relations` regex.
Все метрики пересчитаны для всей истории.

**Эффект:** v2-диаграмма (library) — изолированных узлов стало 0 вместо 4.

---

### 4. Промпты v2 — strict

**Проблема:** промпты v1 содержали инструкции "реальные системы сложнее чем написано"
и "никогда не упрощай". Агенты добавляли Events, Exceptions, Infrastructure
которых не было в требованиях — архитектурные галлюцинации.

**Изменения в backstory агентов:**

| Агент | v1 (expansive) | v2 (strict) |
|---|---|---|
| Analyst | "дополняй что заказчик подразумевал" | "только то что явно написано" |
| Architect | "никогда не упрощай" | "количество блоков определяется требованиями" |
| Architect | — | "каждый enum подключается явной стрелкой" |

Прежние промпты сохранены в `experiments/prompts_v1_expansive.md`.

---

### 5. Трекинг токенов и стоимости

**Что сделано:** после `crew.kickoff()` извлекается `crew.usage_metrics`.
В `meta.json` сохраняются: `prompt_tokens`, `completion_tokens`, `total_tokens`, `cost_usd`.
Стоимость считается через `litellm.cost_per_token()`.
В UI добавлена третья строка метрик с токенами и стоимостью.
В таблицу аналитики добавлены колонки токенов и стоимости.

---

### 6. Исправление рендеринга PNG

**Проблема:** широкие диаграммы обрезались по правому краю.

**Исправление:** в `render_png()` добавлена функция `_fit_to_canvas()` которая вставляет
`scale max 4000 width` после `@startuml` перед отправкой на kroki.io.

---

### 7. Мультиагентный slim-режим

**Что сделано:** добавлен `run_pipeline_slim()` — тот же пайплайн из 5 агентов,
но с явным ограничением контекста через параметр `context` у каждого Task:

```
Task2 (Architect) → context=[task1]         # только анализ
Task3 (Coder)     → context=[task2]         # только архитектура, без анализа
Task4 (Critic)    → context=[task3]         # только PlantUML-код
Task5 (Fixer)     → context=[task3, task4]  # код + замечания
```

В полном пайплайне каждый агент получает ВСЕ предыдущие выводы — контекст накапливается.
В slim — только то что нужно для конкретной задачи.

## Результаты экспериментов

### Эксперимент A: сравнение режимов — Library EN, Gemini Flash, class

| Метрика | 1 агент | 5 агентов v1 | 5 агентов v2 |
|---|---|---|---|
| Время (с) | 48.9 | 130.3 | 79.2 |
| Блоков | 13 | 27 | 17 |
| Связей | 6 | 16 | 8 |
| Типов связей | 5 | 2 | 3 |
| Строк кода | 123 | 238 | 125 |
| Покрытие req (%) | 43.6 | **52.7** | 43.6 |
| Лишних эл. (%) | 23.1 | 18.5 | **5.9** |
| Изолир. узлов | **0** | 8 | **0** |
| Атрибутов | 22 | 53 | 22 |
| Методов | 13 | 25 | 15 |

**Ключевые наблюдения:**
- v1 генерирует вдвое больше блоков за счёт выхода за рамки требований (Events, Exceptions, Infrastructure)
- v2 радикально снижает лишние элементы (23.1% → 5.9%) и полностью устраняет изолированные узлы
- Одиночный агент показывает больше типов связей (5 vs 2-3) — использует нотацию разнообразнее
- v2 работает на 40% быстрее v1 — строгие промпты генерируют меньше промежуточного текста

| Метрика | 1 агент | 5 агентов v2 |
|---|---|---|
| Время (с) | **28.8** | 86.1 |
| Блоков | 21 | 22 |
| Покрытие (%) | **25.0** | 17.4 |
| Лишних эл. (%) | 4.8 | **4.5** |
| Изолир. узлов | 2 | **0** |

**Ключевое наблюдение:** на структурированных требованиях (Use Case формат) результаты
почти идентичны. Мультиагентный пайплайн не даёт преимущества — структура уже есть
во входных данных, Analyst не добавляет ценности.

### Эксперимент C: сравнение стоимости — Process Sale, GPT-5, class

| Метрика | 1 агент | 5 агентов full | 5 агентов slim |
|---|---|---|---|
| Prompt tokens | 1 998 | 264 200 | 76 128 |
| Completion tokens | 7 123 | 190 128 | 141 984 |
| **Total tokens** | **9 121** | **454 328** | **218 112** |
| **Стоимость** | **$0.07** | **$2.23** | **$1.52** |
| Время (с) | **79.5** | 495.6 | 466.6 |
| Блоков | 50 | **56** | 23 |
| Покрытие (%) | **43.1** | 42.8 | 35.2 |
| Лишних эл. (%) | 4.0 | 1.8 | **0.0** |
| Изолир. узлов | 9 | **2** | 8 |
| Кратность связей (%) | 0 | 20 | **60.5** |

**Ключевые наблюдения:**

**Стоимость:** мультиагентный full в 32x дороже одиночного ($2.23 vs $0.07).
Slim экономит 52% токенов относительно full (218k vs 454k), но всё равно в 22x дороже single.
Причина: в sequential CrewAI каждый агент получает полный контекст всех предыдущих —
к 5-му агенту prompt накапливает 264k токенов.

**Время:** slim почти не быстрее full (466 vs 495с). Узкое место — не размер контекста,
а сама скорость генерации GPT-5 и 5 последовательных API-вызовов.

**Качество:** GPT-5 single даёт сопоставимое качество при радикально меньшей стоимости.
50 блоков, 43.1% покрытие против 56 блоков, 42.8% у full — разница незначительная.

## Сводные выводы

### 1. Ценность мультиагентности зависит от формата входных данных

На **неструктурированных** требованиях (живой текст) мультиагентный пайплайн снижает
процент лишних элементов в 4 раза (23.1% → 5.9%). Analyst структурирует требования,
Architect формирует архитектуру — Coder получает готовый к кодированию ввод.

На **структурированных** требованиях (Use Case, таблицы) разница минимальна.
Одиночный агент справляется так же хорошо.

### 2. Сильные модели нивелируют преимущество пайплайна

GPT-5 single ($0.07) генерирует диаграмму сопоставимого качества с GPT-5 multi ($2.23).
Для слабых моделей (Gemini Flash) пайплайн даёт более заметный выигрыш в точности.

### 3. Накопление контекста — главное ограничение CrewAI

В sequential process контекст растёт с каждым агентом. Для GPT-5:
prompt_tokens = 264k → стоимость = $2.23 за один запуск.
Slim-режим сокращает prompt-токены на 71%, но completion-токены — только на 25%.
Время генерации почти не меняется — узкое место в латентности API и GPT-5, не в размере контекста.

### 4. Промпты критически важны

Смена backstory аналитика и архитектора с "expansive" на "strict" снизила
процент лишних элементов с 18.5% до 5.9% без изменения архитектуры системы.
Это показывает что формулировка промптов влияет на результат сильнее чем выбор модели.

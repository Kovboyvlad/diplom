# Промпты v1 — "Expansive" (расширяющие требования)

Первая версия промптов агентов. Агентам явно давалась инструкция выходить за рамки требований и дополнять то, что "заказчик подразумевал но не сказал". Результат: диаграммы с enterprise-архитектурой поверх простых требований.

---

## Analyst — backstory

```
Ты опытный бизнес-аналитик. Твой главный принцип: реальные системы всегда сложнее
чем написано в требованиях. Ты всегда дополняешь анализ тем, что заказчик подразумевал
но не сказал явно:
- если есть статус объекта — ты выписываешь все его значения как перечисление
- если есть действие — ты определяешь метод с сигнатурой
- если есть взаимодействие — ты описываешь интерфейс
- если упомянута ошибка или исключение — ты описываешь сценарий полностью

Ты никогда не упрощаешь и не объединяешь то, что должно быть раздельным.
```

**Goal:**
```
Извлечь из требований полную структуру системы: все сущности с атрибутами и типами,
все перечисления статусов, все события, сценарии ошибок и неявные компоненты.
```

---

## Architect — backstory

```
Ты главный архитектор с экспертизой в UML и SysML. Твои принципы:
- каждый статус или тип — это отдельный enum, не строка
- каждое внешнее взаимодействие проходит через явный интерфейс
- связанные компоненты группируются в подсистемы
- тип каждой связи указывается явно: не "связан", а "композиция" или "агрегация"
- ты не объединяешь разные сущности в одну ради упрощения

Ты проектируешь столько блоков, сколько реально нужно системе —
никогда не упрощаешь архитектуру в угоду краткости.
```

**Goal:**
```
Спроектировать детальную архитектуру: все блоки с атрибутами и методами,
точные типы всех связей, группировка по подсистемам, enum для каждого статуса,
интерфейсы для внешних взаимодействий.
```

## Наблюдаемые результаты

Тест-кейс: `01_library` (система управления библиотекой)
Модель: `gemini/gemini-3-flash-preview`
Тип диаграммы: `class`

### Мультиагентный пайплайн (5 агентов, v1-промпты)

| Метрика | Значение |
|---|---|
| Время генерации | 130.3 с |
| Блоков | 27 |
| Связей | 16 |
| Типов связей | 2 |
| Строк кода | 238 |
| Покрытие req | 52.7% |
| Лишних элементов | 18.5% |
| Изолированных узлов | 8 |
| Атрибутов | 53 |
| Методов | 25 |

**Что сгенерировано сверх требований:**
- `package "Library.Events"` — 4 event-класса (BookReturnedEvent, FineGeneratedEvent, BookAvailableForReservationEvent, LoanDueDateApproaching)
- `package "Library.Exceptions"` — 5 классов исключений (ReaderBlockedException, CopyUnavailableException и др.)
- `package "Library.Infrastructure"` — INotificationService, IPaymentGateway, NotificationLog, SystemSettings
- `package "Library.Services"` — CatalogService, LoanService, ReservationService, FineService

Ни одно из этих пакетов не упомянуто в исходных требованиях. Пайплайн достроил enterprise-архитектуру самостоятельно.

**Проблемы нотации:**
- Enum-ы подключены только как типы атрибутов (`status: BookCopyStatus`), без явных UML-стрелок → 5 enum-ов считаются изолированными
- `CatalogService` и `LoanDueDateApproaching` не имеют связей вообще
- Используется только 2 типа стрелок (`-->`, `..>`) из всего арсенала UML

---

## Вывод

Промпты v1 (expansive) провоцируют агентов на **архитектурные галлюцинации**: пайплайн добавляет Events, Exceptions, Infrastructure и Service Layer, которых нет в задании. Это не ошибка генерации — это прямое следствие инструкции "реальные системы всегда сложнее чем написано".

Мультиагентный пайплайн с v1-промптами генерирует **больше элементов** (27 vs 13 блоков), но **хуже связывает их** (8 vs 1 изолированных узлов) и **использует беднее нотацию** (2 vs 5 типов связей).

Следующая версия промптов (v2 — "strict") должна ограничить агентов строгим следованием требованиям.

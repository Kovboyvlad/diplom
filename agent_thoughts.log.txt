Внешние актёры (люди и организации)
- Кассир (Primary Actor): инициирует и ведёт продажу, вводит/корректирует позиции, принимает оплату, выполняет отмену/приостановку/возобновление чека, запрашивает чеки и документы.
- Покупатель: предъявляет товары/услуги, предоставляет идентификацию для скидок/кредита, выбирает и осуществляет способ оплаты, подписывает кредитную оплату, может просить отменить/исключить позиции/подарочный чек.
- Продавец (Salesperson): заинтересован в корректном учёте комиссий по продажам.
- Налоговые органы (Government Tax Agencies): получают начисленные налоги с каждой продажи (в т.ч. множественные юрисдикции).
- Компания (владелец POS): заинтересована в корректном учёте, обновлениях инвентаря, устойчивости к сбоям и удовлетворённости клиентов.

Внешние ИТ-системы
- Сервис авторизации платежей (Payment Authorization Service): принимает запросы авторизации по кредитным/дебетовым платежам и возвращает решение (approve/deny).
- Внешняя бухгалтерская система (Accounting): принимает данные о продаже и оплате для финансового учёта и расчёта комиссий.
- Внешняя система учёта запасов (Inventory): принимает данные о проданных позициях для списания/резерва остатков.
- Внешний сервис расчёта налогов (Tax Calculation Service): по входным данным о позиции/чеке возвращает рассчитанные налоги.

Система под разработку (POS) — логические блоки
- UI слой POS (сенсорный экран): интерфейс для кассира/покупателя (отображения сумм, подсказок, захват подписи).
- Блок аутентификации и сессий: идентификация и авторизация кассира, восстановление сессии.
- Менеджер продажи (Sale Management): создание/ведение чека, позиции, статусы (активен/приостановлен/отменён/завершён), логирование, восстановление после сбоя.
- Каталог и ценообразование (Pricing): поиск товара по идентификатору, описание, базовая цена, применение набора правил ценообразования.
- Движок бизнес‑правил (Pluggable Rules Engine): вставляемые правила на шагах 3 (позиции/цены/скидки) и 7 (оплата/валидации).
- Блок скидок и льгот (Discounts & Eligibility): проверка статусов (сотрудник, привилегированный клиент и т.п.), расчёт скидок по правилам.
- Блок клиентского кредита/баланса (Customer Credit): учёт и применение накопленного кредита клиента к сумме продажи.
- Обработчик купонов (Coupons): приём и валидация купонов, уменьшение цены, учёт использованных купонов.
- Налоговый модуль/адаптер (Tax Engine/Adapter): расчёт налогов локально или через внешний Tax Calculation Service; обработка сбоев/ручного ввода.
- Платёжный модуль (Payments):
  - Подмодуль наличных (Cash): ввод суммы, расчёт сдачи, управление денежным ящиком.
  - Подмодуль кредит/дебет (Card): захват реквизитов карты, взаимодействие с Payment Authorization Service, обработка approve/deny, захват подписи.
  - Подмодуль чеков (Check): приём и запись параметров чека (детали не раскрыты).
- Интеграция с бухгалтерией (Accounting Integration): упаковка и отправка сведений о продаже/оплате, комиссиях.
- Интеграция с инвентарём (Inventory Integration): отправка сведений о проданных позициях для обновления запасов.
- Генератор документов (Documents & Receipts): печать стандартного чека, подарочного чека (без цен), форм и квитанций по ребейтам.
- Журналирование и восстановление (Logging & Recovery): запись транзакционно значимых событий/состояний, реконструкция после сбоев, фиксация ошибок.
- Международзация (i18n): локализация текста интерфейса.
- Подсистема устройств (Device Controllers): сканер штрих‑кода, клавиатура, картридер/пин‑пад, подпись (бумажная/цифровая), денежный ящик, принтер чеков.

Физические устройства ввода/вывода
- Сканер штрих‑кода / клавиатура: ввод идентификаторов товаров, сумм, данных клиента.
- Картридер / клавиатура: ввод реквизитов кредитного/дебетового счёта.
- Механизм захвата подписи: бумажный чек или электронная подпись (планируется переход на цифровую).
- Денежный ящик: выдача наличных/сдачи под управлением системы.
- Принтер чеков/форм: печать чека, подарочного чека, форм ребейтов.

Границы и контракт взаимодействия
- POS является центральным оркестратором. Все внешние взаимодействия (налоги, авторизация, бухгалтерия, инвентарь) инициируются POS.
- Все действия кассира и покупателя проходят через UI POS и соответствующие доменные блоки (правила, налоги, платежи).
- Транзакционно значимые состояния (позиции, итоги, оплаты, статусы чека) журналируются для восстановления.

Предусловия
- Кассир идентифицирован и аутентифицирован в POS. Открыта сессия кассира и готов UI.

Постусловия (гарантия успеха)
- Продажа сохранена; налоги рассчитаны корректно; бухгалтерия и инвентарь обновлены; комиссионные записаны; чек/квитанции сгенерированы; авторизации по платежам сохранены.

Основной поток взаимодействий (UC1: Process Sale)
1. Покупатель подходит к кассе с товарами/услугами.
   - Покупатель -> Кассир: передаёт товары/сообщает услуги.
   - Кассир -> POS(UI): инициирует новую продажу.
   - POS(Sale Management): создаёт черновик продажи, начинает журналирование.

2. Кассир начинает новую продажу.
   - POS отображает пустой чек, готов к вводу позиций.

3. Ввод позиции.
   - Кассир -> Сканер/клавиатура -> POS(UI): вводит идентификатор товара (UPC/EAN/JAN/SKU, сканер предпочтительно).
   - POS(Pricing): находит товар, применяет набор ценовых правил.
   - POS(Sale Management): добавляет строку продажи.

4. Подтверждение позиции.
   - POS(UI): показывает описание товара, цену по правилам, промежуточный итог.
   - Повтор шагов 3–4, пока кассир не завершит ввод позиций.

5. Расчёт и показ итогов.
   - POS(Tax Engine): рассчитывает налоги (в т.ч. через внешний Tax Calculation Service).
   - POS(UI): показывает общий итог с налогами, разбивку при необходимости.

6. Озвучивание суммы и запрос оплаты.
   - Кассир -> Покупатель: сообщает итог и запрашивает оплату.
   - Покупатель выбирает способ оплаты.

7. Обработка оплаты.
   - Ветвление по способу оплаты (см. альтернативы 7a–7e):
     - Наличные: ввод суммы, расчёт сдачи, открытие денежного ящика, запись платежа.
     - Кредит/дебет: захват реквизитов карты, запрос авторизации во внешний сервис, обработка approve/deny, захват подписи, запись платежа с токеном авторизации.
     - Чек: обработка согласно правилам (не детализировано).
     - Купоны: до проведения оплаты, регистрация и применение купонов, запись использованных купонов.
     - Клиентский кредит: применение кредита до нуля.
   - POS обновляет статус чека.

8. Завершение и интеграции.
   - POS(Logging): логирует завершённую продажу.
   - POS(Accounting Integration): отправляет данные продажи и оплаты во внешнюю бухгалтерию (в т.ч. для комиссий).
   - POS(Inventory Integration): отправляет информацию по позициям для списания/обновления остатков.

9. Генерация чека/документов.
   - POS(Documents): печатает обычный чек; по условиям — формы ребейтов; по запросу — подарочный чек.

10. Завершение обслуживания.
   - Покупатель получает чек и товары и покидает кассу.
   - Продажа помечена как завершённая.

Альтернативные потоки и условия (ветвления и взаимодействия)

Общий сбой системы (в любой момент) — *a
- Условие: системная ошибка POS или узла.
- Действия:
  1) Кассир перезапускает POS, входит в систему, запрашивает восстановление.
  2) POS(Recovery) реконструирует предыдущее состояние по журналу.
  2a) Если обнаружены аномалии:
      - POS сигнализирует ошибку; пишет запись об ошибке; переводит систему в чистое состояние.
      - Кассир начинает новую продажу.

Шаг 3: Ввод позиции — частные случаи
- 3a. Неверный идентификатор:
  - POS сигнализирует ошибку кассиру и отвергает ввод. Взаимодействие возвращается к шагу 3.
- 3b. Много одинаковых товаров (уникальная идентичность не важна):
  - Кассир вводит идентификатор категории и количество.
  - POS добавляет позицию с указанным количеством, обновляет итог.

Шаги 3–6: Корректировки по просьбе покупателя
- 3–6a. Удаление позиции:
  - Кассир вводит идентификатор для удаления.
  - POS удаляет строку и показывает обновлённый итог.
- 3–6b. Отмена продажи:
  - Кассир отменяет чек в POS.
  - POS помечает чек как отменённый, логирует событие.
- 3–6c. Приостановка продажи:
  - Кассир приостанавливает чек.
  - POS сохраняет чек для последующего извлечения на любом терминале.

Шаг 4: Несогласие с автоматически рассчитанной ценой — 4a

  1a) Если не восстановлено:

      - Кассир либо вручную рассчитывает и вводит налог, либо отменяет продажу.

- Покупатель заявляет право на скидку.
- Кассир помечает запрос скидки и вводит идентификацию покупателя.
- POS(Discounts) рассчитывает скидку по правилам и показывает новую сумму.

  - 1b) Просит отменить продажу → кассир отменяет чек в POS.

Шаг 7: Оплата наличными — 7a
- Кассир вводит внесённую сумму.
- POS рассчитывает сдачу и открывает денежный ящик.
- Кассир кладёт деньги и выдаёт сдачу.
- POS записывает платёж наличными.

Шаг 7: Оплата кредитной картой — 7b

  - 3a) Получен отказ:

- 5–6) POS показывает механизм захвата подписи; кассир запрашивает подпись; покупатель подписывает (бумажно или цифрово).

Шаг 7: Оплата чеком — 7c
- Обработка чека согласно политике (ввод реквизитов, возможная валидация). Детали не раскрыты, POS фиксирует платеж.

Шаг 7: Оплата дебетовой картой — 7d
- Аналогично кредитной с авторизацией через внешний сервис и захватом PIN/подписи по правилам.

Шаг 7: Купоны — 7e
- До проведения платежа кассир регистрирует каждый купон.
- POS уменьшает цену согласно правилам и записывает использованные купоны для учёта.
- 1a) Купон не соответствует купленным товарам:
  - POS сигнализирует ошибку кассиру; купон не принимается.

- 9a. Ребейты:
  - POS генерирует формы и квитанции по товарам с ребейтами.
- 9b. Подарочный чек:

Действия (что делают) и Логика взаимодействия (как обмениваются сообщениями)
- Главный поток (Basic Flow)
  1) Покупатель -> Кассир: приносит товары/услуги к покупке.
  2) Кассир -> POS-Приложение: начинает новую продажу.
     - POS создаёт объект продажи (статус: черновик).
  3) Кассир -> (Сканер/Клавиатура) -> POS: вводит идентификатор товара.
  4) POS -> Каталог/Прайсинг/Правила: находит товар, рассчитывает цену по правилам; POS -> UI: добавляет линию продажи, описание, цену, промежуточный итог.
     - Повтор шагов 3-4, пока кассир не завершит ввод.
  5) POS -> Налоговый сервис: рассчитывает налог; при успехе POS -> UI: показывает итог с налогами.
  6) Кассир -> Покупатель: озвучивает итог; Кассир -> Покупатель: запрашивает способ оплаты.
  7) Оплата:
     - Если наличные:
       a) Кассир -> POS: вводит внесённую сумму.
       b) POS -> Денежный ящик: отпирает; POS -> UI: показывает сдачу.
       c) Кассир -> Покупатель: вносит наличные в ящик, возвращает сдачу.
       d) POS: записывает платёж наличными.
     - Если кредитная карта:
       a) Покупатель -> Кард-ридер/Клавиатура -> POS: вводит реквизиты карты.
       b) POS -> Сервис авторизации: отправляет запрос на авторизацию.
       c) Сервис авторизации -> POS: возвращает одобрение или отказ.
       d) При одобрении: POS записывает платёж с данными одобрения; POS -> Подпись: показывает механизм ввода подписи; Кассир -> Покупатель: запрашивает подпись; Покупатель вводит подпись.
       e) При отказе: POS -> Кассир: сигнализирует отказ; Кассир -> Покупатель: запрашивает альтернативную оплату.
     - Если чек/дебет: POS следует соответствующим сценариям оплаты (аналогично кредиту, с соответствующими интеграциями/правилами).
     - Купоны (если есть) — до оплаты: Кассир -> POS: регистрирует каждый купон; POS: валидирует и уменьшает цену, записывает использованные купоны; при несоответствии товара купону — ошибка кассиру.
     - Скидки (если заявлены): Кассир -> POS: помечает запрос скидки; Кассир -> POS: вводит идентификацию клиента; POS -> Правила скидок: рассчитывает и показывает сумму скидки.
     - Кредит на счёте клиента (если есть): Кассир -> POS: запрашивает применение кредита; Кассир -> POS: вводит идентификатор клиента; POS: применяет кредит до нулевой цены и уменьшает остаток кредита.
  8) POS -> Бухгалтерия: отправляет данные продажи и оплаты (для учёта и комиссий).
     POS -> Инвентарь: отправляет данные для обновления остатков.
     - При недоступности: Модуль интеграций буферизует и повторяет отправку до успеха.
  9) POS -> Принтер: печатает чек; при наличии ребейтов — печатает формы/квитанции; при запросе подарочного — печатает подарочный чек без цен.
  10) Покупатель: уходит с чеком и товарами.

- Альтернативные/расширенные потоки
  A) Сбой системы в любой момент:
     - Кассир: перезапускает POS, входит в систему, запрашивает восстановление.
     - POS Модуль восстановления: реконструирует предыдущее состояние продажи.
     - Если обнаружены аномалии: POS сигнализирует ошибку, логирует инцидент, очищает состояние; Кассир начинает новую продажу.
  B) Шаг 3 — неверный идентификатор:
     - POS: сообщает об ошибке и отклоняет ввод; Кассир повторяет ввод.
  C) Шаг 3 — множественный ввод однотипных товаров:
     - Кассир -> POS: вводит идентификатор категории и количество (например, 5 упаковок); POS добавляет агрегированную линию.
  D) Шаги 3-6 — покупатель просит удалить позицию:
     - Кассир -> POS: вводит идентификатор позиции к удалению; POS пересчитывает и показывает новый итог.
  E) Шаги 3-6 — отмена продажи по просьбе покупателя:
     - Кассир -> POS: отменяет продажу; POS помечает чек как отменённый.

     - Кассир -> POS: приостанавливает; POS сохраняет так, чтобы чек был доступен для продолжения на любой кассе (статус: приостановлен).
  G) Шаг 4 — цена по умолчанию не подходит (ценовой override):
     - Кассир -> POS: вводит переопределённую цену (с правами/причиной); POS показывает новую цену (аудитирует override).

  I) Шаг 5 — скидка по статусу клиента:
     - Кассир -> POS: инициирует скидку; вводит ID клиента; POS применяет по правилам и показывает итог со скидкой.
  J) Шаг 5 — кредит на счёте клиента:
     - Кассир -> POS: инициирует применение кредита; вводит ID клиента; POS уменьшает сумму до нуля максимально возможным кредитом и списывает кредит.
  K) Шаг 6 — не хватает наличных:
     - Покупатель: выбирает альтернативный способ оплаты
       или просит отменить продажу; Кассир отменяет в POS.
  L) Шаг 7 — оплата наличными (детализация):
     - Кассир: вводит сумму; POS: показывает сдачу, открывает ящик; Кассир: выдаёт сдачу; POS: записывает платёж наличными.
  M) Шаг 7 — оплата кредитной картой (детализация):
     - Покупатель: вводит реквизиты; POS -> Сервис авторизации: запрос.
     - Если нет связи: POS сигнализирует кассиру; Кассир просит альтернативную оплату.
     - Если одобрено: POS записывает платёж и данные одобрения; POS инициирует захват подписи; Кассир запрашивает подпись; Покупатель подписывает.
     - Если отказ: POS сигнализирует; Кассир просит альтернативу.
  N) Шаг 7e — купоны:

  O) Шаг 9a — ребейты:
     - POS определяет позиции с ребейтами; POS -> Принтер: печатает формы ребейтов и квитанции.
  P) Шаг 9b — подарочный чек:
     - Кассир -> POS: запрашивает; POS печатает чек без цен.

2) Главные внутренние блоки POS (контейнеры) и ответственность
- POS Orchestrator (Процесс Продажи / Workflow-движок):
  - Главный координатор сценариев UC1.
  - Управляет состоянием продажи, шагами, устройствами и внешними вызовами.
  - Обеспечивает, что сервер (оркестратор) управляет клиентом (UI) и устройствами.
- UI/Touch Frontend:
  - Экран кассира; ввод SKU/количеств/скидок/ID клиента/оплаты; отображение итогов/налогов/сдачи/ошибок.
  - Интернационализация, крупный шрифт.
  - Не содержит бизнес-логики, вызывает команды оркестратора и подписывается на его события.
- Sale Domain Service (Модуль продажи):
  - Модель чека: линии, суммы, статусы (черновик, приостановлен, завершён, отменён).
  - Правила инвариантов, пересчёт итогов, фиксация событий домена.
- Pricing & Catalog Service (Каталог/Прайсинг):
  - Поиск товара по UPC/EAN/JAN/SKU; базовая цена, описание.
  - Поддержка ценового override с аудитом.
  - Кэш каталога локально; синхронизация вне UC1 — вне рамок этого кейса, но предусмотрено.
- Rules Engine (Плагинные бизнес-правила):
  - Пайплайн правил ценообразования (шаг 3/4) и скидок/купонов/кредитов/правил оплаты (шаг 7).
  - Расширяемый реестр правил; конфигурируемый порядок применения; версионирование.
- Discounts/Promotions Service (Купоны/Скидки/Кредиты):
  - Валидация купонов, применение скидок (в т.ч. по статусу клиента), применение клиентского кредита.
  - Взаимодействует с Rules Engine.
- Tax Module (Модуль расчёта налогов):
  - Адаптер к внешнему Tax Calculation Service.
  - Политика отказоустойчивости; fallback на ручной ввод налога кассиром с явным подтверждением и аудитом.
- Payments Service (Модуль оплаты):
  - Сценарии наличные/карта/чек/дебет.
  - Интеграция с Payment Authorization Service.
  - Запись платёжных событий, авторизаций; управление захватом подписи.
- Devices Manager (Подсистема локальных устройств):
  - Драйверы/адаптеры: сканер, кард-ридер/пин-пад, денежный ящик, принтер, модуль подписи.
  - Единый API устройств; очереди команд; управление состоянием устройств.
- Printing Service (Модуль печати):
  - Печать обычного чека, подарочного чека, форм/квитанций ребейтов.
  - Шаблоны, макеты, локализация.
- Integration Gateway (Шлюз интеграций):
  - Асинхронная доставка в Accounting и Inventory.
  - Буферизация (outbox), повторные попытки, дедупликация, мониторинг.
- Journal & Recovery (Журналирование и восстановление):
  - Транзакционно-чувствительный журнал событий и снапшотов.
  - Восстановление прерванной продажи/сессии; фиксация аномалий.
- AuthN/AuthZ (Аутентификация кассира и права):
  - Идентификация кассира; контроль прав (override цены, отмена и т.п.).
- Persistence (Локальная БД и кэши):
  - Таблицы: Sales, SaleLines, Payments, Authorizations, CouponsUsed, TaxBreakdown, EventsLog, Outbox, DevicesState, Users/Roles.
  - ACID для локальной фиксации продажи и outbox.
- Config/Secrets & I18N:
  - Управление конфигурацией, ключами интеграций, словарями локализации.

6) Жизненный цикл продажи (состояния и переходы)
- Черновик (Draft): создан при StartSale.

    - Suspend -> Приостановлен (Suspended).
    - Cancel -> Отменён (Cancelled).

- Приостановлен: доступен к возобновлению на любой кассе (Resume -> Draft).
- Завершён: иммутабельный; печать чека/подарочного; публикация в Outbox.
- Отменён: иммутабельный; инвентарь/бухгалтерия не обновляются как продажа (возможна запись отмены).

8) Последовательности (укрупнённые)
- Basic Flow (3-7-9):
  - UI StartSale -> Orchestrator -> Sale Domain create Draft.
  - UI ScanItem -> Orchestrator -> Catalog/Pricing + Rules -> Sale Domain add line -> TotalsUpdated.
  - Repeat.
  - Orchestrator -> Tax Module (sync) -> TaxCalculated -> TotalsUpdated.
  - UI ChoosePayment:
    - Cash: PayCash -> Orchestrator -> Devices(OpenDrawer) -> Sale Domain record payment -> Completed.

  - Resume -> Recovery loads sale -> state to Draft.
- Failure on Tax:
  - Tax Module timeout -> Orchestrator -> UI prompt manual tax -> cashier enters -> Sale Domain updates -> continue.

  - method, amount, changeAmount?, status, details.
- Authorization:
  - authCode, providerRef, responseCodes, maskedPAN/token, expiry (не хранить PAN).
- Tax:
  - perLine breakdown, jurisdictions, method=auto/manual, confirmerId.
- Coupon/Discount:
  - code, type, amount/percent, validationResult, linkage to lines.
- Инварианты:
  - Сумма оплат = grandTotal (за вычетом change).
  - Налог либо auto, либо manualConfirmed=true.
  - Авторизации карточных платежей обязательны для card methods.
  - Любое изменение состава/цен/скидок инвалидирует предыдущий расчёт налога до пересчёта/подтверждения.

10) Надёжность и устойчивость
- Журналирование и восстановление:
  - Event Sourcing light: ключевые события домена + периодические снапшоты продажи.
  - При старте POS: Recovery предлагает восстановить последнюю сессию/продажу.
- Интеграции:
  - Outbox-таблица с атомарной записью вместе с завершением продажи.
  - Ретрай с экспоненциальной задержкой; дедупликация по idempotencyKey.
  - Локальная очередь выдерживает недоступность внешних систем; продажа завершается локально.
- Таймауты:
  - Tax: короткие таймауты, 2–3 ретрая, далее fallback.
  - Payments: SLA ≤30 сек для 90%; таймаут => ошибка взаимодействия и запрос альтернативного метода.
- Печать и устройства:
  - Очередь печати; при сбое — повтор, логирование, возможность отложенной печати (ре-импринт).
- Ошибки:
  - Централизованный Error Bus; UI получает понятные коды/сообщения.
  - Аудит всех overrides/ручных вводов с userid/причиной/меткой времени.

11) Устройства и их абстракция
- Device Drivers (плагины): по интерфейсу IDriver: init, status, execute(command), events.

  - Управляет жизненным циклом, очередями команд, приоритезацией (напр. открытие ящика блокирует до подтверждения).
